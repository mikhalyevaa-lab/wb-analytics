## Блок A — Хранение (wb_storage_daily)

### Известные проблемы
- `barcodes_count` и `cost_per_unit` — NULL на всех строках до перезагрузки в этой сессии
- Март 2026: до перезагрузки данные были завышены в 4286% (barcodesCount × warehousePrice)
- Текущая формула `cost = warehousePrice` даёт результат ~14x ниже Еженедельного отчёта — причина неизвестна
- Все месяцы янв–июн 2026 перезагружены с правильной формулой в этой сессии

### A1 — Верификация покрытия после перезагрузки ✋ Ждёт подтверждения
**Задача:** проверить что все дни 01.01.2026–20.06.2026 покрыты без пропусков  
**Критерий приёмки:** 171 уникальных дней в wb_storage_daily для данного store_id  
**Метод:** `SELECT COUNT(DISTINCT date) FROM wb_storage_daily WHERE store_id = '...'`

### A2 — Верификация barcodes_count после перезагрузки ✋
**Задача:** убедиться что barcodes_count теперь не NULL (код обновлён, данные перезагружены)  
**Критерий:** `SELECT COUNT(*) FROM wb_storage_daily WHERE barcodes_count IS NULL` = 0 для строк после Jan 2026  
**Если NULL — нужна ещё одна перезагрузка**

### A3 — Сверка хранения с Еженедельным отчётом ✋
**Задача:** сравнить `SUM(cost) FROM wb_storage_daily` за каждую неделю vs `storage_cost` из `wb_weekly_reports`  
**Ожидаемое:** расхождение ~14x (известная проблема, WB включает доп.сборы)  
**Цель:** задокументировать точный коэффициент расхождения и его причины  
**Вывод:** показывать пользователю предупреждение что данные API хранения ≠ данные отчёта WB

### A4 — Исследование причины расхождения × 14
**Гипотезы:**
1. WB billing включает НДС (20%) — множитель 1.2, недостаточно
2. warehousePrice — цена за объём (litre), а не за строку — требует уточнения документации WB
3. WB считает платное хранение по другой формуле для коробов и паллет
4. В paid_storage API отсутствуют некоторые строки (приёмка, услуги)

**Метод:** взять конкретный nmId и неделю, сравнить сумму из API построчно с суммой из детализированного отчёта  
**Критерий приёмки:** найдена формула с отклонением ≤5% от отчёта

## Блок E — Подсказки к полям (Hint-компоненты)

### E1 — Блок "Хранение" (/storage)
| Поле | Подсказка |
|---|---|
| Итого за период | Сумма `cost` из `wb_storage_daily` за выбранный период. Не включает НДС и ряд сборов — может расходиться с еженедельным отчётом WB |
| Среднее в день | Итого ÷ количество дней в периоде |
| Залежи | SKU с нулевыми заказами за последние 28 дней при наличии остатка на складе WB |
| Индекс хранение/выручка | cost ÷ выручка × 100%. Норма: < 3%. Красная зона: > 10% |
| Данные по | Дата последней успешной загрузки из WB API paid_storage |

### E2 — Блок "Финансовые отчёты" (/finance-reports)
| Поле | Подсказка |
|---|---|
| Продажа | Сумма выкупленных товаров по розничным ценам WB за отчётный период |
| К перечислению за товар | Сумма к выплате продавцу ПОСЛЕ вычета комиссии WB, до вычета услуг логистики и хранения |
| Итого к оплате | Финальная сумма выплаты: К перечислению − Логистика − Хранение − Штрафы ± Корректировки |
| Прочие удержания/выплаты | Удержания за нарушения, самовыкупы, штрафы СЦ, возвраты |
| Корректировка ВВ | Перерасчёт вознаграждения Wildberries за предыдущие периоды |
| Сверка | Сравнение данных сводного отчёта с суммами по строкам детализации. Допуск: ≤0.5% |

### E3 — Дашборд (/dashboard)
| Поле | Подсказка |
|---|---|
| Выкупы | Продажи с is_realization = true из wb_sales. Исключены возвраты и отмены |
| Заказы | Все строки wb_orders или wb_sales включая будущие выкупы и отмены |
| % выкупа | Выкупы ÷ Заказы × 100% за период. Рассчитывается по данным wb_sales |
| Средний чек | Сумма finished_price ÷ количество выкупов |

### E4 — Блок продаж (wb_sales)
| Поле | Подсказка |
|---|---|
| forPay | Сумма к перечислению продавцу за конкретную продажу: цена со скидкой − комиссия WB |
| finishedPrice | Итоговая цена покупателя с учётом всех скидок и СПП |
| priceWithDisc | Цена товара с учётом скидки продавца (без СПП) |
| SPP | Скидка постоянного покупателя — дополнительная скидка от WB покупателям с историей |
| discountPercent | Скидка продавца, установленная в личном кабинете |

## Блок F — Взаимосвязи данных между блоками

### F1 — Карта источников данных
```
wb_sales (API / XLSX)
  └── те же продажи ←→ wb_weekly_report_rows (тип документа = Продажа)
  └── nm_id → products (справочник)

wb_storage_daily (API paid_storage)
  └── storage_cost за неделю ←→ wb_weekly_reports.storage_cost (с расхождением)
  └── nm_id → products

wb_weekly_reports (сводный XLSX)
  └── содержит агрегаты: Продажа, Логистика, Хранение, К перечислению
  └── ←→ wb_weekly_report_rows (детализация = источник агрегатов)

wb_weekly_report_rows (детализированный ZIP)
  └── for_pay_seller = К перечислению по строке
  └── delivery_service_cost = Логистика по строке  
  └── row_storage_cost = Хранение по строке
```

### F2 — Кросс-чеки в UI
**На странице /finance-reports/[номер]:**
- Показывать: "Хранение по wb_storage_daily за период: X ₽ (расхождение с отчётом: Y%)"
- Показывать: "Продажи в wb_sales за период: Z шт, X ₽"

**На странице /storage:**
- Показывать: "По Еженедельному отчёту WB за период: X ₽" (если отчёт загружен)

**На дашборде:**
- KPI "Выкупы ₽" → ссылка-пояснение на источник данных

### F3 — Единый справочник товаров
**Задача:** убедиться что все nm_id в wb_sales, wb_storage_daily, wb_weekly_report_rows  
присутствуют в таблице `products`  
```sql
SELECT DISTINCT nm_id FROM wb_sales WHERE nm_id NOT IN (SELECT nm_id FROM products)
```
**Критерий:** 0 nm_id без записи в products
