# ABC-анализ — Спецификация (Блок N)

## Executive Summary

Страница `/abc` позволяет владельцу магазина на WB быстро найти убыточные и застойные товары для вывода, а также увидеть «звёзды» ассортимента. ABC считается на лету по двум осям: выручка и маржинальность — с настраиваемыми порогами и произвольным периодом.

---

## Problem Statement

Сейчас у пользователя нет быстрого способа понять, какие из 134 SKU «кормят» магазин, а какие тихо поедают деньги на хранение. Ручной разбор в Excel занимает несколько часов. Цель — найти C-товары и принять решение: вывести, снизить закупку или оптимизировать цену.

## Success Criteria

- За < 30 секунд пользователь видит список C-товаров с отрицательной маржой
- Таблица загружается < 2 секунд
- Пороги настраиваются без перезагрузки страницы

---

## User Journey

1. Открывает `/abc`, видит сводку: A — 12 SKU (1.4 млн ₽), B — 28, C — 94
2. Нажимает «C» в фильтре — таблица фильтруется
3. Сортирует по «Маржа%» по возрастанию — видит товары с -20%
4. Кликает на строку — всплывает правая панель с динамикой выручки и остатками
5. Видит: «был A 30 дней назад, стал C» (стрелка ↓) — принимает решение вывести

---

## Functional Requirements

### P0 — Must Have

**N1 — Расчёт ABC по выручке (ABC-R)**
- Берётся `SUM(wb_sales.finished_price)` за выбранный период по каждому `nm_id`
- Товары сортируются по убыванию выручки, накопительная сумма делит на A/B/C
- Порог A% (дефолт 80), B% (дефолт 95 = 80+15), C% (остаток)

**N2 — Расчёт ABC по маржинальности (ABC-M)**
- `margin = SUM(for_pay) - SUM(cost_price * qty)` за период
- `margin_pct = margin / SUM(finished_price) * 100`
- ABC-M: A = margin_pct ≥ порог_A (дефолт 30%), B = ≥ порог_B (дефолт 10%), C = < порог_B или отрицательная

**N3 — Сводка (карточки)**
```
[A: N SKU / XXX ₽]  [B: N SKU / XXX ₽]  [C: N SKU / XXX ₽]
```
- Отдельно для ABC-R и ABC-M (два ряда или переключатель)

**N4 — Таблица товаров**
Колонки: фото (28×28), Название + nm_id, Выручка ₽, Выручка% от итого, Маржа%, ABC-R, ABC-M, Остатки (дней)

**N5 — Фильтры**
- Кнопки: Все / A / B / C — по ABC-R (активный = выбранная ось)
- Переключатель оси: Выручка | Маржа
- Поиск по vendor_code / nm_id

**N6 — Пресеты периода**
Кнопки: 7д | 14д | 30д | 90д | Свой период (date picker)

**N7 — Настройка порогов**
Кнопка «Настройки» → панель со слайдерами:
- ABC-R: порог A (70–90%, дефолт 80), порог B (10–25%, дефолт 15)
- ABC-M: порог A (15–50%, дефолт 30%), порог B (0–15%, дефолт 10%)
Изменение пересчитывает таблицу без перезагрузки.

**N8 — Правая панель (по клику на строку)**
- Мини-заголовок: фото + название + nm_id
- ABC-R и ABC-M текущего периода vs предыдущего аналогичного (стрелка ↑ green / ↓ red / = gray)
- График: выручка по неделям (bar chart, Chart.js)
- Остатки: количество и дней до нуля (на основе avg_orders_per_day из products)

### P1 — Should Have

**N9 — «Кандидаты на вывод» (красная зона)**
Автоматически выделяется / показывается флаг если товар одновременно:
- 0 заказов за 30 дней ИЛИ
- Маржа < 0 ИЛИ
- Остатки > 90 дней при ABC-R = C

**N10 — Экспорт CSV**
Кнопка «Скачать» — текущая таблица (с фильтрами) в .csv

### P2 — Nice to Have

**N11 — Доля выручки от итого (кумулятивная)**
Колонка «% накоп» с прогресс-баром — видно где пролегает граница A/B

## Technical Architecture

### API Route

`GET /api/abc?period=30&thresholdA=80&thresholdB=15`

```ts
// Запрос wb_sales за период, JOIN с products (cost_price, photo_url, avg_orders_per_day)
// JOIN wb_stocks (текущие остатки)
// Расчёт ABC-R и ABC-M на стороне сервера
// Возвращает массив SKU с полями: nm_id, title, photo_url, vendor_code,
//   revenue, revenue_prev, margin_pct, margin_pct_prev,
//   abc_r, abc_m, abc_r_prev, abc_m_prev,
//   stock_qty, days_of_stock
```

Расчёт предыдущего периода: `dateFrom - period` до `dateFrom` — для стрелок сравнения.

### Данные

| Поле | Источник |
|------|---------|
| revenue | `SUM(wb_sales.finished_price)` за период, `sale_id LIKE 'S%'` |
| for_pay | `SUM(wb_sales.for_pay)` за тот же период |
| cost | `products.cost_price * кол-во продаж` |
| margin_pct | `(for_pay - cost) / revenue * 100` |
| stock | `SUM(wb_stocks.quantity_full)` |
| days_of_stock | `stock / products.avg_orders_per_day` |

### Frontend

- Страница: `src/app/(app)/abc/page.tsx`
- Клиентский компонент с `useState` для фильтров, периода, порогов
- Данные из `/api/abc` через SWR с ключом `[period, thresholdA, thresholdB]`
- График в панели: Chart.js (`cdn.jsdelivr.net`)
- Правая панель: `<Sheet>` или `<Drawer>` (Radix или встроенный)

### Производительность

- Товаров < 500 → расчёт на лету в JS (клиент или сервер)
- wb_sales за 90 дней: ~15–20k строк для одного магазина — Supabase справится за < 1с
- Пагинация не нужна (< 500 SKU)

---

## Out of Scope

- Исторические снимки ABC (история = сравнение с пред. периодом)
- ABC по количеству заказов (не запрошено)
- Автоматические действия (вывод / заказ) — только рекомендация
- XYZ-анализ (нестабильность спроса)
