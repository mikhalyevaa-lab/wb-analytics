**Обновлено:** 2026-06-29

---

## О проекте

**WB Analytics** — аналитическая платформа для продавцов на Wildberries.  
Подключается к WB API, загружает данные, рассчитывает метрики и показывает их в едином дашборде.

**Текущая фаза:** MVP в работе. Один магазин, один пользователь → переход к SaaS.  
**Стек:** Next.js 15 App Router + TypeScript + PostgreSQL 17 (Docker) + `postgres.js` + `better-auth` + Vercel.

---

## Ключевые принципы

| Принцип | Суть |
|---------|------|
| **Data-first** | Каждая цифра верифицируется с официальными отчётами WB. Расхождение > 2% — баг. |
| **Правило 5 шагов** | Discovery → ТЗ → Реализация → Тестирование → Приёмка. |
| **Прозрачность расчётов** | Пользователь видит откуда берётся каждая цифра. |
| **Защита данных** | Чувствительные данные скрыты для ролей без доступа — в UI и на уровне API. |

### П-3. Оптимизация загрузки и снижение API-запросов к ВБ *(новая задача)*

**Проблема:** Синхронизация не использует дельта-загрузку — каждый раз перезагружает все данные с нуля. При росте объёма данных это будет дорого (WB API rate-limits, время выполнения).

**Что нужно:**
1. **Delta sync:** для wb_orders, wb_sales — грузить только от `MAX(last_change_date)` вместо всего периода
2. **Дедупликация запросов:** если несколько компонентов запрашивают одни данные — кешировать результат (in-memory или Redis)
3. **Батчинг:** запросы к WB API с разбивкой по стoreId и периоду вместо монолитных запросов
4. **Мониторинг:** логировать кол-во запросов к WB API в день, алерт при превышении лимитов
5. **Документация:** описать текущие лимиты WB API и как мы в них укладываемся

**Критерий готовности:** синхронизация в 2–3× быстрее, кол-во запросов к WB API снижено в 2× без потери данных.

### П-4. Сверка продаж (wb_sales)

**Проблема:** 83 000+ строк из двух источников (API + XLSX), возможны дубли, нет UNIQUE constraint.

**Что нужно:**
1. Проверить дубли: `SELECT sale_id, COUNT(*) FROM wb_sales GROUP BY sale_id HAVING COUNT(*) > 1`
2. Сверить `SUM(finished_price)` vs `sale` из wb_weekly_reports за совпадающий период (допуск ≤2%)
3. Найти пропуски в датах: нет ли периодов > 3 дней без данных
4. Добавить: `ALTER TABLE wb_sales ADD CONSTRAINT wb_sales_store_sale_unique UNIQUE (store_id, sale_id)`

**Критерий готовности:** 0 дублей, данные сверены, constraint добавлен.

1. `warehousePrice` — цена за литр объёма, не за единицу товара
2. WB включает НДС 20% + доп.тарифы (упаковка, приёмка)
3. В API нет строк по транзитным складам

**Что нужно:**
1. Взять конкретный nm_id и неделю — сравнить строки из API с детализацией из ZIP
2. Задокументировать формулу и точный коэффициент
