# Git Workflow

## Структура веток

```
main          ← только через Pull Request, прямой пуш запрещён
  └── develop ← ветка разработки, основа для всех задач
        └── feature/dev-{context} ← ветка конкретной задачи
```

**main** — production-ветка. Изменения попадают только через PR из develop.  
**develop** — ветка разработки. Все задачи стартуют отсюда.  
**feature/dev-{context}** — ветка конкретной задачи, создаётся от develop.

## Именование веток задач

Формат: `feature/dev-{context}`

- Если есть номер задачи: `feature/dev-42`, `feature/dev-123`
- Если номера нет: `feature/dev-user-roles`, `feature/dev-abc-panel`

`{context}` должен быть коротким и понятным — номер задачи или 1-3 слова описания.

## Жизненный цикл задачи

### 1. Начало задачи — создать ветку от develop

```bash
git checkout develop
git pull origin develop
git checkout -b feature/dev-{context}
```

Никогда не начинать задачу прямо в develop или main.

### 2. Коммиты в процессе работы

Коммитить после каждой **рабочей части** — когда код работает и логически завершён.  
Не коммитить сломанный код. Не накапливать всё в один финальный коммит.

Примеры правильных точек для коммита:
- Реализована одна страница или компонент
- Добавлен и проверен API-роут
- Применена миграция БД
- Исправлен конкретный баг

Формат сообщения коммита:
```
тип: краткое описание что сделано

Подробности если нужны (опционально)

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

Типы: `feat` (новая функция), `fix` (баг), `refactor`, `docs`, `migrations`, `infra`

### 3. Завершение задачи — смержить в develop

```bash
git checkout develop
git pull origin develop
git merge feature/dev-{context}
git push origin develop
```

### 4. В main — только через Pull Request

Прямой пуш в main запрещён.  
Изменения в main идут только через PR из develop после проверки.

## Чего нельзя делать

- `git push origin main` напрямую — только через PR
- Начинать задачу от main вместо develop
- Коммитить нерабочий код
- Один большой коммит на всю задачу в конце

## Проверка текущего состояния

```bash
git branch              # текущая ветка
git status              # несохранённые изменения
git log --oneline -5    # последние коммиты
```
